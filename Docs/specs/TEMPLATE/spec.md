# Task Spec: <Title>

## Summary

Describe the user-visible outcome and why this task matters. State the goal, not the implementation.

## Scope

- Include the user-facing behavior, workflows, or outcomes this task must cover.
- Include required documentation, migration, or compatibility outcomes.
- Avoid naming specific implementation files, types, methods, or architecture unless they are fixed constraints.

## Non-Goals

- List related work that is intentionally excluded.
- List decisions that are deferred to another task.

## Requirements

- Define functional requirements as observable behavior.
- Define constraints or compatibility expectations that the solution must satisfy.
- Capture assumptions if the user has not confirmed every detail.

## Constraints

- List product, platform, performance, accessibility, data, privacy, or compatibility constraints.
- Include implementation constraints only when they are externally required or already agreed.
- Do not prescribe internal structure, file changes, class names, algorithms, or UI layout details unless they are part of the goal.

## Acceptance Criteria

- The expected user-visible behavior is complete.
- Edge cases and error states in scope are handled.
- Relevant docs are updated.
